根据您的要求，我设计一个代码完全共享、仅通过配置区分“传”/“送” 的传送门系统。这个方案可以最大程度地减少代码重复，简化维护，并充分利用Godot场景的实例化特性。

一、核心设计理念

“一个脚本，两种状态”：
• 创建唯一的 teleporter.gd 脚本

• 两个场景 (teleporter_in.tscn, teleporter_out.tscn) 都使用这个脚本

• 通过导出变量或场景继承属性区分初始状态

• 运行时可以动态切换状态

二、需要修改/创建的文件清单

1. 新增通用传送门脚本 (必须创建)


teleporter.gd
  └── 继承自Mechanism，包含完整的传送门逻辑


2. 修改现有传送门场景 (需要调整)

• teleporter_in.tscn → 使用teleporter.gd，设置is_entrance = true

• teleporter_out.tscn → 使用teleporter.gd，设置is_entrance = false

3. 修改Switch脚本 (已存在，需扩展)

• Switch.gd → 添加对传送门的控制方法

4. 修改编辑器管理脚本 (已存在，需适配)

• level_editor.py → 适配新的传送门结构

三、工作流程与修改点

阶段1：创建通用传送门脚本


文件：teleporter.gd
工作内容：
1. 定义传送门基类，继承Mechanism
2. 添加核心属性（颜色、状态、配对目标）
3. 实现状态切换、传送、碰撞检测等通用逻辑
4. 通过导出变量区分“传”/“送”初始状态


阶段2：重构传送门场景


文件：teleporter_in.tscn 和 teleporter_out.tscn
工作内容：
1. 将两个场景的脚本都改为teleporter.gd
2. 在场景中设置不同的导出变量初始值
3. 确保场景结构完全一致（仅label文本不同）


阶段3：扩展Switch控制逻辑


文件：Switch.gd
工作内容：
1. 添加连接传送门的方法
2. 实现切换同色传送门状态的逻辑
3. 修改通知机制，包含传送门


阶段4：适配编辑器配对系统


文件：level_editor.py
工作内容：
1. 修改传送门配对逻辑，适应新结构
2. 更新保存/加载传送门状态的方法
3. 调整测试模式下的连接管理


阶段5：实现传送功能


文件：teleporter.gd
工作内容：
1. 添加Area2D碰撞检测
2. 实现主角传送逻辑
3. 添加传送特效和状态验证

八、配置差异示例

两个场景的唯一区别在于场景属性：
# teleporter_in.tscn 的场景属性
is_entrance = true
label_text = "传"

# teleporter_out.tscn 的场景属性  
is_entrance = false
label_text = "送"


但在运行时，两者可以互相切换状态，因为使用完全相同的代码。

九、验证要点

1. 代码一致性检查：确保两个场景使用完全相同的脚本和节点结构
2. 状态切换测试：Switch控制下，“传”和“送”能正确互换
3. 传送功能验证：只有“传”状态的门能传送，目标位置准确
4. 颜色系统测试：不同颜色的传送门独立工作
5. 保存/加载验证：状态和配对关系能正确持久化

这个设计方案实现了您要求的“两个结构，但代码完全一样”，通过场景配置而非代码重复来区分初始状态，既简化了维护，又保持了系统的灵活性。
